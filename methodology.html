How we derived the ERP spec

A reviewable, traceable synthesis of IMDA's Pre-Approval Guide requirements with mainstream ERP vendor knowledge — producing a checklist that's regulator-backed, market-aware, and SME-fit. Last regenerated 2026-04-27 03:14 UTC.

Data lineage — where the numbers come from

Every capability in the spec traces back to one of two inputs — the IMDA PSG guide (Singapore-regulator-backed) and public mainstream ERP vendor documentation. Every use case traces back to one of these capabilities.

Source mix — how much of the spec is regulator-backed

Each capability-level requirement is tagged with its origin. IMDA = the PSG guide cites it; proxy = mainstream ERP vendors cover it but IMDA doesn't test; both = regulator-confirmed and market-standard.

12-step pipeline

Every transformation from raw PSG text to final use case is one named, re-runnable step with its own intermediate JSON. Full technical detail in the developer pipeline view.

  1. Scrape + parse (deterministic)
    89 IMDA sol cats fetched and parsed into structured requirements. Bot-protection handled in the fetcher.
  2. Group into capabilities (Haiku)
    Each requirement is placed into one of ~30 canonical ERP capabilities per module.
  3. Merge IMDA + proxy (Haiku, per capability)
    Duplicate/overlapping requirements are unified; every statement carries a source tag (imda / proxy / both) for audit.
  4. Cluster into features (Haiku)
    Each requirement is assigned to one of 2-6 features per capability — a sub-capability layer for navigation.
  5. Re-rank criticality (Haiku)
    Required / Preferred / Advanced based on merged context — not blindly inherited from either source.
  6. SME minimum rollup (Haiku)
    3-8 stakeholder-ready statements per SME-applicable capability, each with a rationale.
  7. Vendor questions (Haiku)
    One vendor-facing question per capability. Gates checklist visibility.
  8. Sector tagging (deterministic + Haiku)
    Each requirement carries full IMDA sol-cat + vertical provenance; each capability is flagged generic vs sector-specific.
  9. Verification method (Haiku)
    Each spec tagged as screenshot / demo / config_check / not_verifiable.
  10. Use case generation (Haiku, three phases)
    Top-down from IMDA questions, bottom-up from feature combinations (with coverage sweep), and generic module-level rollups. Each UC carries a derivation method.
  11. Re-home + retrofit (Haiku)
    Semantic re-homing to the best primary capability (within or across modules), with mapped-feature retrofits when needed.
  12. Tier validation + audit (Haiku + deterministic)
    Baseline ⊂ Standard ⊂ Advanced cumulative tiers. 11 deterministic audit rules block bad outputs from shipping.

Per-module coverage

Ten ERP modules. Every module is canonicalised the same way, so counts are directly comparable.

Trust & traceability principles

Grounded

Every requirement points to its origin — an IMDA question title, a mainstream vendor, or both. Nothing is generated without a cited source.

Inspectable

Every transformation produces an intermediate JSON file. Steps can be re-run independently and outputs diffed.

Auditable

11 deterministic audit rules run on every build. P0 violations block shipping. Report at pipeline.html.

Reviewable

Stakeholders see the final checklist; reviewers can drill down to the source text on either side of the merge.
