Architecture

ERP Mapper takes Singapore's PSG (Productivity Solutions Grant) Pre-Approval Guide — 89 solution categories across 14 verticals, published on GitBook — and projects them onto a canonical capability library that also draws from mainstream reference ERPs (NetSuite, Salesforce, HubSpot, Odoo, ERPNext, SAP, Oracle, Workday). Today's outputs are stakeholder-ready vendor self-assessment checklists per module, cross-module compliance baselines (CEM, PDPA, VA/PT), 546 typed integration edges between capabilities, and 71 sol-cat audit reports. Future direction extends the same cap library to project synthesised sol cats, pain-point demand sensing, solution-fit scoring, cap-progression maps, and educational flow visualisations.

Stack: Python 3.9+ (httpx, BeautifulSoup, anthropic, pyyaml) for the pipeline. Standalone HTML + rough.js + Mermaid for viewers. Claude Haiku 4.5 for LLM synthesis.

Future direction (locked 2026-04-30)

The cap library is the central abstraction. Everything else is a projection. Five future direction streams:

1. Synthesised sol cats — generate new (non-IMDA) sol cats by composing caps to address a problem statement / business outcome. Output: candidate checklists PSG could publish.

2. Pain-point → cap → solution discovery — inverse direction. Given an unmet pain point, locate addressing cap(s), audit existing solution coverage, surface gap as market signal.

3. Solution → checklist fit-test — given a candidate solution, score it against problem statement (a) and synthesised sol cat (b). Combines #1 + reverse-vendor-scoring.

4. Cap input-format variants — document how a cap can be fulfilled by different input formats (manual / Excel / POS-generated / API-fed) at different maturity stages.

5. Cap-flow visualisation — diagrams showing cap-replacement / advancement / disruption patterns. Visualisation tier of #4.

Sequencing: #4 → #1 → #5 → #3 → #2. Currently scoped only — no new build until module checklist finalisation completes.

Locked architecture decisions (2026-04-30)

1. Cap library is the central abstraction; everything else is a projection over it.

2. Authored vs derived split is enforced by step6c/6d/13 + override files. New future authored files (synthesised_solcats/, pain_points.yaml, cap_input_variants.yaml) follow the same pattern.

3. data/workflows.yaml survives — hand-curated cap flows; foundation for #4 and #5.

4. pipeline/flow_gen/step3_propose_inter_module.py + step4_propose_intra_module.py + output/flow_review.html archived. Proposed-flow generation was a dead end superseded by direction pivot.

5. pipeline/uc_gen/* archived entirely. UC concept fully retired per direction pivot 2026-04-24.

6. Synthesised sol cats use parallel evidence_refs (decision: extend imda_refs schema with optional source_type to handle both IMDA + synthetic).

7. Each future use case (#1-#5) is an agentic workflow — composable Haiku tool-use, idempotent reads-canonical-writes-derived. No multi-step state machines yet.

Run order (operator-level)

> Loops marked xargs -P 4 are safe to run in parallel (independent reads, per-output writes). Drop -P 4 if running on a low-RAM host.

# Per module (×10)
python3 -m pipeline.canonicalise <module_slug>

# Workspace-level (after all modules)
python3 scripts/derive_solcat_modules.py
python3 scripts/build_solcat_sub_module_join.py
python3 -m pipeline.steps.step13_l2_questions_sync
python3 scripts/audit_l2_question_consistency.py  # gate: P0=0
ls output/solcat_to_checklist_gap_*.json | sed 's|.*gap_||;s|\.json||' | xargs -n1 -P 4 -I{} python3 scripts/analyse_solcat_to_checklist_gap.py {}

# Viewers
ls data/proxy/*.yaml | xargs -n1 basename | sed 's/.yaml//' | xargs -n1 -P 4 -I{} python3 scripts/build_canonical_viewer.py {}
python3 scripts/build_stakeholders_viewer.py
python3 scripts/build_vendor_coverage_viewer.py
python3 scripts/build_index_viewer.py
python3 scripts/build_architecture_viewer.py
